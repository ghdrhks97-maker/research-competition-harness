# deep-interview

가재코드의 `deep-interview`처럼, 코드(=보고서 생성)를 시작하기 전에 **의도를 대화로 명확히** 한다. 사용자가 모든 정보를 미리 입력하게 하지 않는다 — **한 번에 하나씩 물어보고, 선택지를 승인창으로 제시하며, 함께 구체화**한다.

## 원칙
- **한 번에 한 질문.** 사용자를 정보 입력폼처럼 대하지 않는다.
- **승인창(질문 UI)으로 선택지를 제시**한다. Claude Code는 `AskUserQuestion` 도구를 쓴다(선택지 2~4개 + "직접 입력"). 이 UI가 없는 앱은 대화 턴으로 같은 질문을 한다.
- **주제가 없으면 에이전트가 제안**하고, 사용자가 고르거나 수정하게 한다.
- 각 답이 확정되면 짧게 요약하고 다음으로 넘어간다.
- 개인정보(실명 등)는 인터뷰에 담지 않는다.

## 인터뷰 순서

1. **참가 연구대회** — "어떤 연구대회에 내시나요?" 모르면 후보를 제시(수업혁신사례 연구대회, 창의교육 연구대회, 교실수업개선 실천사례 연구대회, 과학전람회, 교육자료전 등)하고 성격(개인/팀, 분량, 심사기준)을 확인. 양식·공문 파일이 있으면 경로를 받아 `rch import-rules`.
2. **대상 학급/학생** — 학교급·학년, 인원, 학급 특성(예: 28명 혼성반, 탐구 흥미 낮음).
3. **전공 교과/분야** — 예: 음악, 과학, 초등 통합.
4. **연구 주제** — "생각해두신 연구 주제가 있나요?"
   - 있으면: 받아서 함께 다듬는다(범위·초점·실행 가능성).
   - 없으면: `brainstorm` 에이전트(또는 직접)로 전공·관심 트렌드·2022 개정 핵심역량을 반영한 **주제 후보 3~4개**를 만들어 승인창으로 제시 → 사용자가 선택/수정.
5. **연구 제목** — 확정 주제로 제목 후보(알파벳 약어형·한글 스토리형) 3~5개를 승인창으로 제시 → 선택/수정. 제목·주제는 **2022 개정 6대 핵심역량 중 하나 이상과 반드시 연계**.
6. **목표 역량·활용 도구·제약** — 기르고 싶은 역량, 활용 도구(예: 아이패드·개러지밴드), 차시·기간 등.
7. **수업을 통해 바라는 결과(desired outcomes)** — "이 수업으로 학생에게서 어떤 변화를 기대하나요?"를 구체적으로 도출한다. 3~5개 결과를 문장으로(예: "감상 활동에서 근거를 들어 설명하는 학생 비율 증가", "협업 창작 참여도 향상"). 이 결과가 **예상 설문값의 기준**이 된다.
8. **보유 자료 확인** — 설문 CSV, 수업사진, 우수 보고서(레퍼런스) 유무와 경로.
   - **없어도 진행한다.** 설문이 없으면 `survey-analyst`가 위 "바라는 결과"에 근거한 **예상 설문값(가상)** 을 만들어 보고서를 완성하고, 나중에 실제값으로 교체할 수 있게 안내한다(강요하지 않음). 사진이 없으면 부록에 "사진첨부필요" 자리표시로 진행한다.

## 확정·기록

인터뷰가 끝나면 확정 사항을 `input/ideas/`에 직접 기록한다(파이썬 `rch brainstorm`을 콘텐츠로 쓰지 않는다):
- `input/ideas/00-interview.md` — 대회·대상·교과·역량·도구·제약·보유자료
- `input/ideas/02-research-topics.md` — 확정 주제(+후보)와 2022 핵심역량 연계
- `input/ideas/03-title-candidates.md` — 확정 제목(+후보)
- `input/ideas/05-desired-outcomes.md` — **수업을 통해 바라는 결과 3~5개**(예상 설문값의 기준)
- `input/rules/competition-profile.json` — 대회명·분야 프로필

## 계획 승인 (ralplan 단계)

기록 후 **보고서 계획을 요약**해 보여주고 승인을 받는다(승인창):
- 확정 주제·제목·핵심역량
- 권장 목차(I~V 또는 대회 양식)
- 채울 자료(설문/사진/레퍼런스/배경연구)와 없는 자료 처리 방침

사용자가 승인하면:
1. `input/rules/competition-profile.json`에 **`"plan_approved": true`** 를 기록한다 (autopilot 상태 머신 `rch next`가 이 값을 읽는다).
2. `report-orchestrator`의 **autopilot 루프**로 Phase 1~5(분석·집필·검증·렌더)를 **추가 질문 없이 연속 실행**한다. 이 승인이 마지막 질문이다 — 이후에는 `rch next`가 `needs_user`를 낼 때(동의·개인정보·자료 확보 등)만 다시 묻는다.

승인 전에는 본문 생성을 시작하지 않고, `plan_approved`도 기록하지 않는다.
